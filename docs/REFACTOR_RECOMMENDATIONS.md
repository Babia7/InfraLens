# Refactor & Structure Improvement Ideas

This document captures a targeted refactor backlog after reviewing the InfraLens repository structure and major architecture files.

## Snapshot of Current Shape

- The app is a React + Vite single-page experience with route composition managed by a central registry (`config/toolsRegistry.ts`) and route identifiers in `types.ts`.
- Feature code is concentrated in `components/apps/` with many domain-specific apps.
- Data source files are centralized under `data/` with contract tests already in place.
- Documentation is present (`docs/ARCHITECTURE.md`, `docs/TECHNICAL_DESIGN.md`, etc.) and can be expanded into a stronger contributor workflow.

## High-Impact Refactor Opportunities

### 1) Split the route registry by domain

**Current pain:** `config/toolsRegistry.ts` is a large, single object that mixes routing, ownership hierarchy (`parentId`), audience metadata, and lazy-load configuration.

**Suggestion:**
- Create `config/routes/` and split into domain files:
  - `coreRoutes.ts`
  - `catalogRoutes.ts`
  - `salesRoutes.ts`
  - `technicalRoutes.ts`
- Keep a small `index.ts` that merges route maps and validates duplicate paths/IDs.

**Benefit:** easier ownership boundaries, simpler reviews, reduced merge conflicts, and safer route additions.

---

### 2) Decouple global types from app-domain models

**Current pain:** `types.ts` includes route enums plus many domain model interfaces (design exports, book metadata, roadmap, etc.) in one large file.

**Suggestion:**
- Keep only shared primitives in `types/index.ts`.
- Move domain contracts into `types/domain/` (e.g., `sales.ts`, `fabric.ts`, `content.ts`).
- Optionally move route enum/route-related types into `config/routes/types.ts`.

**Benefit:** faster navigation, cleaner import boundaries, lower cognitive load.

---

### 3) Introduce feature folders for large app modules

**Current pain:** `components/apps/` is flat, with many app components and related "About" variants side-by-side.

**Suggestion:**
- Use feature directories for larger apps, e.g.:
  - `features/storage-planner/`
  - `features/sales-playbook/`
  - `features/protocol-lab/`
- Co-locate component(s), hooks, test(s), and data adapters per feature.

**Benefit:** clearer ownership, easier testing, less accidental coupling between apps.

---

### 4) Add selector/services layer between UI and raw data

**Current pain:** many UIs consume data directly from source files under `data/`.

**Suggestion:**
- Add `services/selectors/` for compositional read APIs (filtering, normalization, derived views).
- Keep `data/` as canonical raw content and contracts only.

**Benefit:** easier future migration to remote data sources and simpler component logic.

---

### 5) Standardize "About" page patterns

**Current pain:** Several tools have separate `*_ABOUT` route IDs and separate about components; others pass `startAbout` props.

**Suggestion:**
- Define one convention:
  - either nested sub-route with shared screen shell, or
  - one component with mode prop, generated by route metadata.
- Add helper factory in route config to create main/about pairs.

**Benefit:** less route/config duplication and fewer inconsistencies.

---

### 6) Harden boundaries with lint rules and path aliases

**Current pain:** architectural intent exists in docs but is not fully enforced.

**Suggestion:**
- Add ESLint import boundary rules (e.g., no direct cross-feature imports outside allowed paths).
- Expand path aliases for `@features`, `@routes`, `@domain-types`, `@selectors`.

**Benefit:** keeps the intended structure healthy over time.

---

### 7) Expand tests from smoke to contract + composition coverage

**Current pain:** useful tests exist, but route and composition invariants can regress silently.

**Suggestion:** add tests for:
- unique route paths and IDs,
- all `SectionType` values registered,
- parent chain validity,
- audience metadata consistency,
- "About" pairing conventions.

**Benefit:** safer refactors with less manual QA overhead.

---

### 8) Improve docs as a contributor workflow

**Current pain:** docs describe architecture but do not fully prescribe "how to add a new app end-to-end".

**Suggestion:**
- Add a `docs/CONTRIBUTOR_WORKFLOW.md` checklist:
  1. create feature files,
  2. add route metadata,
  3. register seed/data contracts,
  4. add tests,
  5. update docs/changelog.

**Benefit:** faster onboarding and consistent change quality.

## Suggested Execution Plan

### Phase 1 (low risk, high return)
1. Split route registry into multiple domain files.
2. Break `types.ts` into domain files.
3. Add route invariant tests.

### Phase 2 (medium risk)
1. Introduce feature folders for 2-3 highest-change apps.
2. Add selectors/service adapters for shared data reads.

### Phase 3 (structural hardening)
1. Enforce import boundaries and aliases.
2. Unify About-page pattern and route factories.
3. Formalize contributor workflow docs.

## Optional Success Metrics

- Avg PR files touched per feature change decreases.
- Route/config regressions caught by tests before review.
- Time-to-add-new-app reduced via workflow checklist.
- Fewer merge conflicts in route and type files.
